1. Executive Summary

AEGIS GCS is currently implemented as a native desktop Ground Control Station shell. The application is built around a Qt Widgets main window, a typed telemetry bus, a thread-safe vehicle state model, a runtime plugin host, and a MAVLink UDP input/output layer. It supports live/dummy telemetry, log replay parsing, plugin panels, alert visualization, mission editor scaffolding, and a real OpenStreetMap-backed native map view.

Current status: AEGIS is a working modular GCS foundation with core runtime services and multiple plugins. It is not yet a complete certified flight-control product, but its architecture is suitable for iterative hardening toward commercial-grade deployment.

5. Core Services

TelemetryBus

Typed Qt signal bus for hot-path telemetry and system events. It emits attitude, position, battery, heartbeat, alerts, connection state, command responses, and outbound vehicle commands.

VehicleState

Thread-safe canonical model of the active vehicle. Uses QReadWriteLock and exposes typed getters/mutators with update signals.

PluginHost

Discovers shared libraries, reads Qt plugin metadata, validates IPlugin, initializes plugins with bus/state/config, and manages unload lifecycle.

Shared Types

Defines telemetry and command data structures: AttitudeData, PositionData, BatteryData, SystemState, and VehicleCommand.

6. Telemetry and MAVLink Pipeline

The live telemetry path receives MAVLink frames over UDP on a dedicated worker thread. Parsed messages are emitted through Qt queued connections to the main-thread parser, which updates VehicleState and broadcasts typed bus events.

UDP Socket → MavlinkIO → MavlinkParser → VehicleState + TelemetryBus → UI/Plugins

Current MAVLink Decoding

MAVLink MessageAEGIS Output
HEARTBEATSystem ID/component, base mode, custom mode, armed state, system status.
ATTITUDERoll, pitch, yaw, angular rates.
GLOBAL_POSITION_INTLatitude, longitude, altitude, relative altitude, velocity, heading.
BATTERY_STATUSRemaining percentage, voltage, current.
MISSION_CURRENTCurrent mission index.
STATUSTEXTAlert severity mapping and alert console output.

Connection Lifecycle

MavlinkIO now separates socket availability from vehicle health. It only emits connected after receiving a MAVLink heartbeat. If no heartbeat arrives within the configured timeout, it emits disconnected. Shutdown stops timers, closes sockets, clears remote state, and waits for the worker thread.

Important design decision: socket bind success is not treated as a vehicle connection. A vehicle is considered alive only after heartbeat evidence.

8. User Interface Shell

The UI shell is a Qt Widgets QMainWindow with menu bar, toolbar, status bar, central map area, and right-side docked plugin panels. The map plugin is injected as the central widget. Other plugins are wrapped in QDockWidget containers and tabified.

Current UI Features

  • Dark operator theme via ThemeEngine.
  • File menu for opening replay logs.
  • Connect menu and toolbar action for UDP MAVLink.
  • Status bar connection indicator.
  • Vehicle armed/mode display derived from heartbeat state.
  • Plugin injection and basic dock/tab management.

9. Map View Implementation

The current map uses Qt native graphics rather than an embedded browser. It converts latitude/longitude to Web Mercator pixel coordinates, downloads OpenStreetMap tiles, places them in a QGraphicsScene, and overlays vehicle position, heading, and track history.

FeatureImplementation
Tile providerConfigurable URL template, defaulting to https://tile.openstreetmap.org/%1/%2/%3.png.
ProjectionWeb Mercator tile coordinate formula.
NetworkingQNetworkAccessManager and QNetworkReply.
RenderingQGraphicsView, QGraphicsScene, pixmap tile items, path overlay, marker overlay.
ZoomChanges OSM tile zoom level and refreshes tile set.
Operational note: OSM tile usage requires respecting OpenStreetMap tile policy. A commercial deployment should add tile caching, provider attribution, rate limiting, and preferably a dedicated commercial tile provider or offline MBTiles support.

10. Replay and Test Infrastructure

The replay service reads binary MAVLink frames from log files and emits decoded messages into the normal parser path. This allows replayed telemetry to update the same state/bus/UI pipeline as live UDP telemetry.

11. Build, Deployment, and CI

The project builds with CMake and Qt6. Core libraries are static targets; plugins are shared libraries deployed under the runtime plugins/ directory. On Windows, the build integrates windeployqt for Qt runtime DLL deployment.

cmake -B build -S . -DCMAKE_PREFIX_PATH="C:/Qt/6.8.2/msvc2022_64" -DAEGIS_BUILD_TESTS=ON
cmake --build build --config Release --parallel
ctest --test-dir build --output-on-failure -C Release

GitHub Actions currently defines Linux and Windows build/test jobs using Qt installation action, CMake, and CTest.
