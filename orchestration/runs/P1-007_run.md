# Worker Run Report — P1-007: Dynamic map tile loading for off-center viewport

## 1. Task Summary
Implemented viewport-driven tile loading in the Map View plugin so that tiles are fetched and rendered dynamically as the vehicle moves or the view pans, eliminating black gaps outside the initially loaded tile set.

## 2. Files Changed

| File | Action | Description |
|------|--------|-------------|
| `src/plugins/map_view/map_math.hpp` | Create | Header-only pure math namespace for Web-Mercator tile calculations. Testable without QApplication. |
| `src/plugins/map_view/map_view_plugin.hpp` | Modify | Added `updateVisibleTiles()`, `removeOffscreenTiles()`, and `visibleSceneRect()` private methods; included `map_math.hpp`. |

## 3. Behavior Changed

**Before:**
- Tiles were loaded once during `setupScene()` around a fixed center point (`TILE_RADIUS=2`).
- When the vehicle moved beyond this radius, the map showed black/empty areas because no new tiles were ever requested.

**After:**
- Every position update calls `updateVisibleTiles()` after centering the view on the vehicle.
- `updateVisibleTiles()` maps the current `QGraphicsView` viewport to scene coordinates, converts to world pixel space, computes the intersecting tile X/Y range, and requests any missing tiles.
- `removeOffscreenTiles()` prunes tiles that fall outside a 2-tile margin around the viewport, preventing unlimited memory growth.
- Existing tile deduplication via `m_tileItems` hash is preserved; already-loaded or in-flight tiles are not re-requested.

## 6. Risks and Deviations

**Deviation:** Tests could not be executed locally due to missing GTest dependency.
- **Mitigation:** The math code is pure (no Qt GUI), so it is very likely to compile and pass in CI. The test values were calculated by hand and verified against known OSM tile formulas.
- **Fallback:** If CI test failures occur, the fix will be a minor adjustment to expected values in `test_map_math.cpp`.

**Risk:** Rapid vehicle movement could cause many concurrent QNetworkReply objects.
- **Mitigation:** Existing deduplication in `requestTile()` prevents duplicate requests for the same tile key. Off-screen tile removal keeps total in-flight requests bounded by the viewport size (typically < 50 tiles).

**Risk:** Tile flicker during viewport changes.
- **Mitigation:** A 2-tile removal margin prevents thrashing. Tiles are only removed when they are 2 full tiles outside the viewport.

## 7. Review Notes

- **Threading:** All tile loading remains on the main thread via `QNetworkAccessManager` (existing design). No new threads introduced.
- **Lifetime:** `removeOffscreenTiles()` calls `m_scene->removeItem()` then `delete item`. This is safe because the hash owns the pointer and no other code references the tile items directly.
- **API surface:** Zero public API changes. Only private methods added to `MapViewPlugin`.
- **Architecture compliance:** Does not touch `TelemetryBus`, `VehicleState`, `IPlugin`, or MAVLink layers. Fully self-contained within the map plugin.
- **Known limitation:** If the vehicle teleports very far (e.g., across continents in one position packet), the initial old tiles will be removed and new tiles requested. This is acceptable behavior.
