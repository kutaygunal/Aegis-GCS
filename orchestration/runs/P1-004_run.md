# Worker Run Report — P1-004: Config Schema Validation

## 1. Task Summary

Extracted the ad-hoc config normalization logic from `application.cpp` into a dedicated `ConfigValidator` class with a declarative schema DSL. The validator validates types, ranges, and allowed values; warns on unknown keys (preserving them for forward compatibility); and fills sensible defaults for missing keys. Added `configVersion` to the config file as the foundation for future migration support.

## 3. Behavior Changed

**New features:**
- `ConfigValidator::createAegisValidator()` returns a pre-built validator for all AEGIS config domains.
- `ConfigValidator::validate()` produces structured warnings with key-path context (e.g., `telemetry.bindPort: expected int, got QVariantMap; using default 14550`).
- Missing optional keys are silently filled with defaults.
- Missing required keys are filled with defaults AND produce a warning.
- Unknown keys are preserved (forward compatibility) but warned.
- Out-of-range values are clamped to min/max with warnings.
- Invalid enum values fall back to default with warnings.
- `configVersion` key added to config schema (default 1, optional).

**Changed behaviors:**
- `Application::initialize()` now creates a `ConfigValidator` and validates config before any service initialization.
- `Application::loadConfiguration()` no longer performs normalization; it just parses JSON.
- All inline `defaultConfig()`, `validateAndNormalizeConfig()`, `mergeVariantMaps()` removed from `application.cpp`.

**Removed behaviors:**
- None.

## 6. Risks and Deviations

**Deviations from plan:**
- None.

**Challenges:**
- MSVC struggled with deeply nested `QMap<QString, SchemaRule>` brace initialization in `createAegisValidator()`. Fixed by building QMaps step-by-step with `insert()` calls.
- Initial `coerceToType()` had unused parameters. Fixed with `[[maybe_unused]]`.
- Empty config test initially failed because missing object-type keys with nested schema but no explicit default returned invalid QVariant. Fixed by recursively validating an empty map against nested schema when default is invalid.

**Known issues / follow-up:**
- SchemaRule uses `QMap<QString, SchemaRule>` for nested rules. For very deep nesting this could be replaced with a pointer-based tree, but current depth (2-3 levels) is acceptable.
- Array element validation is not yet implemented (no array-of-objects config domains exist today).
- No runtime schema editing; schemas are compile-time constants.
